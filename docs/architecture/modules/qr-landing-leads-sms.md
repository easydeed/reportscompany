# Module: QR Landing Pages, Lead Routing & SMS Notifications

---

## Overview

Every property report gets a unique, scannable QR code that links to a branded landing page. When a prospect scans the code (or opens the link), they see property details and an agent contact form. Submitting the form creates a **lead** in the database and (when configured) fires an **SMS text message** to the listing agent in real time.

```
 ┌─────────────┐       ┌──────────────────┐       ┌──────────────┐
 │  QR Code    │──────▶│  Landing Page     │──────▶│  Lead Stored │
 │  (R2 PNG)   │       │  /p/{short_code}  │       │  + SMS Sent  │
 └─────────────┘       └──────────────────┘       └──────────────┘
```

---

## 1. QR Code Generation

### When it happens

A QR code is generated automatically when a property report is created (inside the `POST /v1/property/reports` handler).

### How it works

| Step | Detail |
|------|--------|
| Short code | A unique 8-character `short_code` is generated by a DB trigger on the `property_reports` table. |
| URL encoded | The QR encodes `https://trendyreports.io/p/{short_code}`. |
| Image generation | Uses the `qrcode` Python library with **rounded modules** (`RoundedModuleDrawer`) and **high error correction** (30%) for a modern, reliably-scannable look. Foreground color defaults to black but can be customized per report. |
| Upload | The PNG is uploaded to **Cloudflare R2** at key `qr-codes/{report_id}.png` with a 1-year cache header. |
| Fallback | If R2 is not configured, falls back to an external service (`api.qrserver.com`). |

### Regeneration

`POST /v1/property/reports/{report_id}/regenerate-qr` creates a **new** short code and QR image. The old link immediately stops working.

### Key files

| File | Role |
|------|------|
| `apps/api/src/api/services/qr_service.py` | QR generation, R2 upload, deletion |
| `apps/api/src/api/routes/property.py` | Calls `generate_qr_code` during report creation |

### Environment variables

| Variable | Purpose |
|----------|---------|
| `R2_ACCOUNT_ID` | Cloudflare R2 account |
| `R2_ACCESS_KEY_ID` / `R2_SECRET_ACCESS_KEY` | R2 credentials |
| `R2_BUCKET_NAME` | Bucket (default `market-reports`) |
| `R2_PUBLIC_URL` | Public URL prefix for QR images |

## 2. Landing Page

### Route

`/p/[code]` — a Next.js dynamic page at `apps/web/app/p/[code]/page.tsx`.

### Data fetching

The page calls `GET /v1/property/public/{short_code}` (public, no auth required). The API returns property details, agent info, and landing-page controls.

### What the visitor sees

1. **Hero section** — gradient background using the report's accent color, company logo, full property address.
2. **Property stats** — beds, baths, sqft, year built (conditionally shown).
3. **Agent card** — photo (or initial avatar), name, company, phone (tap to call), email (tap to email), website link.
4. **Lead capture form** — name\*, email\*, phone, message, consent checkbox.
5. **Footer** — "Powered by TrendyReports" branding.

### Status handling

| HTTP Status | Scenario | UI |
|-------------|----------|----|
| `200` | Active report | Full landing page |
| `404` | Short code not found | Next.js not-found page |
| `410` | Inactive, expired, or max leads reached | "Page Not Available" / "Page Expired" status screen |
| `403` | Access code required or IP blocked | "Access Code Required" status screen |

### Analytics

Each page visit updates `view_count`, `unique_visitors`, and `last_viewed_at` on the `property_reports` row.

### Key files

| File | Role |
|------|------|
| `apps/web/app/p/[code]/page.tsx` | Server component — data fetch, hero, agent card, status screens |
| `apps/web/app/p/[code]/lead-capture-form.tsx` | Client component — form with validation, submission, success/error states |

## 3. Lead Capture & Routing

### Endpoint

`POST /v1/leads/capture` — **public**, no authentication. Called directly from the landing page form.

### Anti-spam pipeline (in order)

| # | Check | Failure behavior |
|---|-------|-----------------|
| 1 | **Honeypot** — hidden `website` field must be empty | Silent 200 (fools bots) |
| 2 | **Report lookup** — find report by `short_code` | 404 |
| 3 | **Active check** — `is_active` must be true | 410 |
| 4 | **Expiry check** — `expires_at` must be in the future (or null) | 410 |
| 5 | **Max leads** — current lead count vs `max_leads` cap | 410 |
| 6 | **Access code** — if report has one, payload must match | 403 |
| 7 | **IP block list** — `blocked_ips` table (with optional expiry) | 403 |
| 8 | **Rate limit** — max 5 submissions per IP per hour (`lead_rate_limits` table) | 429 |
| 9 | **Duplicate email** — same email + same report = silent 200 | Silent 200 |

### Lead storage

After all checks pass, a row is inserted into the `leads` table:

| Column | Source |
|--------|--------|
| `account_id` | From the property report |
| `property_report_id` | From the property report |
| `name`, `email`, `phone`, `message` | From the form submission |
| `source` | `"qr_scan"` or `"direct_link"` |
| `consent_given` | Checkbox value |
| `status` | Starts as `"new"` |
| `ip_address`, `user_agent` | From the HTTP request |
| `sms_sent_at` | Set after successful SMS delivery |

### Lead lifecycle

```
new  ──▶  contacted  ──▶  converted
```

Agents manage lead status from the dashboard at `/app/leads` or from individual report detail pages.

### Lead management endpoints (authenticated)

| Method | Endpoint | Purpose |
|--------|----------|---------|
| `GET` | `/v1/leads` | List leads (filterable by status, report) |
| `GET` | `/v1/leads/{id}` | Single lead detail |
| `PATCH` | `/v1/leads/{id}` | Update status / notes |
| `DELETE` | `/v1/leads/{id}` | Remove lead |
| `GET` | `/v1/leads/export/csv` | CSV export of all leads |

### Key files

| File | Role |
|------|------|
| `apps/api/src/api/routes/leads.py` | All lead endpoints — capture, list, update, delete, CSV export |
| `apps/web/app/app/leads/page.tsx` | Agent-facing leads dashboard |
| `apps/web/app/app/property/[id]/page.tsx` | Report detail page (includes associated leads) |

## 4. SMS Text Notifications

### Property report lead flow (API-side)

When a lead is captured, the system checks three conditions before sending an SMS:

1. **Twilio is configured** — environment variables are set.
2. **Account has SMS credits** — `accounts.sms_credits > 0`.
3. **Agent has a valid phone number** — looked up from the `users` table, formatted to E.164.

If all three are met, an SMS is sent **to the agent** via Twilio:

```
🏠 New Lead Alert!

{lead_name} is interested in:
{property_address}
📱 {lead_phone}
📧 {lead_email}

Respond quickly for best results!
- TrendyReports
```

After delivery:
- `accounts.sms_credits` is decremented by 1.
- `leads.sms_sent_at` is set to `NOW()`.

### Consumer report flow (Worker-side)

A separate SMS flow exists for the agent lead page (`/cma/{agent_code}`). When a consumer requests a home value report:

1. The Celery worker task `process_consumer_report` generates the report.
2. The worker sends the report link **to the consumer** via SMS.
3. A notification is also sent **to the agent**.
4. Both messages are logged in the `sms_logs` table with the `consumer_report_id`.

### Twilio integration

SMS is sent via Twilio's REST API using `httpx` (async HTTP client) — no Twilio SDK dependency.

| Component | File |
|-----------|------|
| API (lead capture SMS) | `apps/api/src/api/services/twilio_sms.py` |
| Worker (consumer report SMS) | `apps/worker/src/worker/sms/send.py` |

Both implement:
- `send_sms(to_phone, message)` — core send function
- `send_lead_notification_sms(...)` — formatted agent alert (API only)
- `format_phone_e164(phone)` — normalizes 10-digit, 11-digit, and international numbers
- `is_configured()` — checks for required env vars

### Environment variables

| Variable | Required by | Purpose |
|----------|-------------|---------|
| `TWILIO_ACCOUNT_SID` | API + Worker | Twilio account identifier |
| `TWILIO_AUTH_TOKEN` | API + Worker | Twilio auth token |
| `TWILIO_PHONE_NUMBER` | API + Worker | "From" number in E.164 format (e.g. `+15551234567`) |

## 5. Database Tables

### `property_reports` (relevant columns)

| Column | Type | Purpose |
|--------|------|---------|
| `short_code` | `varchar(20)` | Unique code for QR URL (DB trigger) |
| `is_active` | `boolean` | Landing page toggle |
| `expires_at` | `timestamptz` | Optional expiration |
| `max_leads` | `integer` | Optional lead cap |
| `access_code` | `varchar(50)` | Optional gate code |
| `view_count` | `integer` | Total page views |
| `unique_visitors` | `integer` | Distinct IP views |
| `last_viewed_at` | `timestamptz` | Last visit timestamp |

### `leads`

| Column | Type | Purpose |
|--------|------|---------|
| `id` | `uuid` | Primary key |
| `account_id` | `uuid` | FK to accounts |
| `property_report_id` | `uuid` | FK to property_reports |
| `name`, `email`, `phone`, `message` | `text` | Contact info |
| `source` | `text` | `qr_scan` or `direct_link` |
| `consent_given` | `boolean` | GDPR/TCPA consent |
| `status` | `text` | `new` / `contacted` / `converted` |
| `sms_sent_at` | `timestamptz` | When agent was notified |
| `email_sent_at` | `timestamptz` | When email notification was sent |
| `ip_address` | `text` | Client IP |
| `user_agent` | `text` | Browser user-agent |
| `notes` | `text` | Agent notes |

### `lead_rate_limits`

Tracks submissions per IP for the 5-per-hour rate limit.

### `blocked_ips`

Permanent or time-limited IP blocklist with optional `expires_at`.

### `sms_logs`

Logs for the consumer report SMS flow (worker-side). Stores `consumer_report_id`, `to_phone`, `message`, `status`, timestamps.

## 6. End-to-End Flow

```
1. AGENT creates property report
   ├── DB trigger generates short_code
   ├── QR code PNG generated → uploaded to R2
   └── URL: https://trendyreports.io/p/{short_code}

2. PROSPECT scans QR (or clicks link)
   ├── GET /v1/property/public/{short_code}
   ├── Landing page renders: hero, stats, agent card, form
   └── view_count / unique_visitors updated

3. PROSPECT submits lead form
   ├── POST /v1/leads/capture (public)
   ├── Anti-spam pipeline: honeypot → active → expiry → max leads
   │   → access code → IP block → rate limit → duplicate email
   ├── Lead row inserted into leads table (status: "new")
   └── If Twilio configured + credits > 0 + agent has phone:
       ├── SMS sent to agent with lead details
       ├── sms_credits decremented
       └── sms_sent_at set on lead

4. AGENT manages leads
   ├── Dashboard: /app/leads (list, filter, search)
   ├── Update status: new → contacted → converted
   ├── Add notes
   └── Export CSV
```

## 7. Agent CMA Lead Page (`/cma/{agent_code}`)

Every agent gets a permanent, branded URL for self-serve home-value leads:

```
trendyreports.io/cma/{agent_code}
```

### Address-first funnel (redesigned)

The flow gives value before asking for contact info:

```
Step 1: Address Search  →  Google Places autocomplete + SiteX lookup
Step 2: Property Confirm →  Beds/baths/sqft displayed, "Is this your property?"
Step 3: Contact Capture  →  Phone (SMS) or Email, name (optional), consent
Step 4: Processing       →  Animated progress + report delivery
Step 5: Success          →  "Report sent" + agent CTA buttons (Call / Email)
```

### Dual delivery

| Method | How it works |
|--------|--------------|
| **SMS** (default) | Twilio sends report link to homeowner's phone |
| **Email** | Report link delivered by email (Resend integration planned) |

### Unified lead storage

CMA leads are written to **both** tables:
- `consumer_reports` — tracks the generated report (property data, comps, PDF)
- `leads` — unified lead table with `source = 'cma_page'` and `consumer_report_id` FK

### Agent customization (dashboard at `/app/lead-page`)

| Setting | Where |
|---------|-------|
| Agent code (3-20 chars, customizable) | Lead Page settings |
| Headline / subheadline | Lead Page settings |
| Theme color | Lead Page settings |
| Active / inactive toggle | Lead Page settings |
| Logo, photo, company | From account branding |

### Key files

| File | Role |
|------|------|
| `apps/web/app/cma/[code]/page.tsx` | Server component — branded hero, agent card |
| `apps/web/app/cma/[code]/cma-funnel.tsx` | Client component — 3-step address-first funnel |
| `apps/web/app/app/lead-page/page.tsx` | Agent dashboard — settings + leads list |
| `apps/web/app/app/lead-page/lead-page-client.tsx` | Client — URL/QR display, settings panel, lead rows |
| `apps/api/src/api/routes/lead_pages.py` | Public API — agent info, property search, report request |
| `apps/api/src/api/routes/me.py` | Authenticated API — lead page settings CRUD |
| `apps/api/src/api/services/agent_code.py` | Agent code generation, lookup, validation |
| `apps/worker/src/worker/tasks.py` | `process_consumer_report` — comps, value estimate, SMS/email delivery |

### API endpoints (public, no auth)

| Method | Endpoint | Purpose |
|--------|----------|---------|
| `GET` | `/v1/cma/{agent_code}` | Agent info for landing page |
| `POST` | `/v1/cma/{agent_code}/search` | Property search via SiteX |
| `POST` | `/v1/cma/{agent_code}/request` | Submit report request (phone or email) |

### Database tables

- `consumer_reports` — full report data (property, comps, value estimate, PDF)
- `leads` — unified lead with `source = 'cma_page'`, `consumer_report_id`
- `sms_logs` — SMS delivery tracking
